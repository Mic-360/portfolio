# ShashinMori: The Unauthorized Biography of My Existential Crisis With Photo Storage

> "Why did I spend weeks building what Google should've done?" — A retrospective.

## The Problem Nobody Asked For

You know that feeling when your 2016 Pixel phone still works perfectly fine, but Google Photos decides _you_ don't deserve unlimited original-quality storage anymore? Yeah, that.

Google had this magical offer: unlimited storage for Pixels, _exclusively_ for Pixels. Then they were like, "lol jk," and now you're stuck with 15GB (shared with Gmail, because why not make it even more painful?) unless you subscribe to the Google One torture-mill.

So, naturally, the solution was to:

1. Build a full-stack photo gallery
2. Learn Flutter (because web was too easy)
3. Set up a backend API with job queues
4. Add image processing
5. Configure Firebase authentication
6. Deploy it all somewhere
7. Wonder why I didn't just buy cloud storage

Seemed reasonable at 2 AM.

## The Architecture Nobody Needed

### The Frontend: Flutter, Because I Hate Myself

Let me tell you about Flutter. It's this magical framework where you write once and deploy to web _and_ Android simultaneously. In theory.

In practice, you:

- Write Dart (a language that looks like TypeScript had a baby with Java)
- Deal with BuildContext nightmares
- Discover Riverpod (state management that actually doesn't make you want to scream)
- Realize that GoRouter is basically React Router's better-looking cousin
- Cry when images don't load due to CORS on the first try
- Celebrate when they eventually do

The app features:

- **Google Sign-In**: Because firebase_auth isn't fun enough on its own
- **Responsive Gallery**: Staggered grid that adapts from phone to desktop (and tries very hard not to look broken)
- **Dark Mode**: Because light mode is for people who enjoy eye pain
- **Image Caching**: With shimmer loading states, because we're fancy
- **Real-time Upload Status**: Polls the backend every 500ms like a neurotic person checking their exam results

### The Backend: Fastify, Because Node.js Needed To Be Faster

Fastify is basically Express but someone hit it with the "more performance" stick. Hard.

What did I build?

- **tus Protocol Integration**: So uploads can resume mid-failure (nobody asked for this level of reliability, but here we are)
- **Redis Caching**: Because hitting the database every time is for amateurs
- **Image Processing**: With Jimp, because FFmpeg was too powerful
- **Job Queue**: BullMQ + Redis to handle image processing asynchronously
- **Firebase Auth Verification**: Checking JWT tokens like a paranoid bouncer
- **Swagger Docs**: So future me can remember what this API does
- **Rate Limiting**: Because the internet has bad actors

The tech stack:

## The Experience: From "This Is Cool" to "Why Does Everything Hurt"

### Week 1: Optimism

- "I'll just wire up authentication real quick"
- Firebase setup goes smoothly
- Flutter hello world compiles

### Week 2: Reality

- tus protocol is weirdly specific about chunk handling
- Image processing on constrained devices is... constrained
- CORS errors are my new best friend
- Why does Flutter run slower on web than native?

### Week 3: Acceptance

- Added debugging statements that log to shadows
- Discovered that 50% of tech problems are just "did you turn it off and on again?"
- Image caching hit a 50% ratio and I celebrated like I'd just won a Grammy

### Week 4: Deployment Terror

- Docker containers that work locally but explode in production
- Redis connection pooling? Never heard of her
- Scaling to 5 concurrent users required divine intervention

## The Features You'll Never Use

- **Unlimited Uploads**: Until you hit disk space, then it's unlimited problems
- **Real-Time Sync**: Updates appear in 0.5-3 seconds, depending on your luck
- **Secure Google Auth**: Because passwords are 2010s anyway
- **Beautiful Dark Mode**: Your retinas will thank you
- **Responsive Design**: Works on your phone, tablet, laptop, and that old iPad collecting dust

## What I Learned (The Hard Way)

1. **File uploads are hard**: Why is every protocol either ancient or overcomplicated? tus was the least painful, which speaks volumes.

2. **State management matters**: Riverpod feels like witchcraft but in a good way. Context doesn't scale past about 3 providers.

3. **Image processing is IO-bound**: So offload it to workers and cache aggressively.

4. **Firebase is both a blessing and a curse**: Easy auth = yes. Pricing model = existential dread.

5. **Flutter web is weird**: It works, but it feels like running Android in an emulator running in your browser.

6. **Testing would've been nice**: Narrator: _It would have been._

## The Numbers That Mean Nothing

- **30+ npm packages**: Approximately 47 of which I use
- **5 Firebase services**: Auth, Firestore (looked at but didn't use), Storage (same), Hosting, and regret
- **2 platforms**: Web and Android (iOS is basically "contributions welcome" territory)
- **1 existential crisis**: Completely unrelated to the project
- **~0 users**: My mom reviewed the design, does that count?

## In Conclusion

ShashinMori is what happens when someone looks at "unlimited original quality storage" and decides the solution is to become a full-stack developer.

Is it overkill? Absolutely.
Is it cooler than `rsync` to a NAS? Undeniably.
Will Google release a better version next month? Probably.

But for now, at least when my Pixel finally dies, I'll have a sick portfolio project and a codebase that makes me question every decision I made.

**Would I recommend building this?** Only if you also hate yourself and have strong feelings about unused storage quotas.

**Would I do it again?** Not without therapy.

**Can you use it?** It's open source, so technically yes. Will you? Probably not. Are there security implications I haven't considered? We live in hope.
